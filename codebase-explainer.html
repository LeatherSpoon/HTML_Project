Processing Power Codebase Map

This is a browser-based Three.js idle RPG. The code is organized as a small custom game engine: one coordinator, a Three.js scene layer, gameplay systems, entities, and DOM-based UI.

Runtime Flow

HTML Shell

index.html defines the canvas, HUD panels, combat overlay, mobile controls, styles, import map, and module entry point.

Coordinator

js/main.js creates systems, wires keyboard/click handlers, switches zones, handles interactions, and runs the animation loop.

Frame Loop

gameLoop() updates player, systems, combat, gathering, portals, HUD, camera, and renderer once per frame.

Startup Walkthrough

1. Browser loads

index.html creates all fixed DOM surfaces up front: the WebGL canvas, HUD container, game panels, combat overlay, and mobile controls.

2. Modules resolve

The import map points three at the local vendored Three.js file. This is why the app must be served over HTTP instead of opened with file://.

3. main.js boots

main.js constructs all gameplay systems first, then constructs scene/world/player/UI objects that depend on those systems.

4. Loop starts

sceneManager.renderer.setAnimationLoop(gameLoop) hands control to the browser's animation cadence.

Ownership Boundaries

Scene

SceneManager owns Three.js scene, renderer, orthographic camera, resize handling, camera follow, and render.

Environment builds zones, terrain, props, portals, stations, collision circles, trees, rocks, and spawn metadata.

Entities

Player owns mesh, position, movement, facing, steps, terrain effects, and node gathering state.

EntityManager owns resource nodes and enemies, including aggro-triggered combat starts.

Systems

Small state machines for PP, stats, combat, inventory, crafting, drones, pedometer, saves, achievements, minigame, drill, ascension, and offline rewards.

Input And UI Paths

Input

Keyboard: main.js maintains a keysDown set.

Mobile: TouchInput exposes dx, dz, isMoving, and actionPressed.

Player: Player.update(keysDown, delta, touchInput) consumes both paths and moves the character.

UI

HUD renders most non-combat panels: stats, inventory, crafting, drones, equipment, pedometer, achievements, ascension, drill, minigame.

CombatUI renders the combat overlay, action buttons, skill/item menus, HP bars, FP ring, and combat log.

System Responsibilities

PPSystem

Main idle number: total PP, PP/sec, prestige bonus, temporary modifiers, ascension multiplier.

StatsSystem

Stat levels and derived combat/movement values: HP, FP, energy, damage, defense, agility, speed.

CombatSystem

Combat state, enemy/player HP, FP spends, skills, status effects, drops, run outcomes.

InventorySystem

Materials, consumables, tools, storage container, and equipment bag.

CraftingSystem

Recipes, crafting timers, queueing, material consumption, item/tool output.

DroneSystem

Idle material gathering through assignable/upgradable drones.

PedometerSystem

Step currency, PP-per-step upgrades, track placement, stat purchases, zone unlocks.

SaveSystem

Turns live game state into JSON and applies JSON back to existing system instances.

Where Behavior Lives

Good separation

  • PP math is mostly in PPSystem.
  • Stat derivation is mostly in StatsSystem.
  • Combat mechanics are mostly in CombatSystem.
  • Three.js setup is mostly in SceneManager.

Mixed responsibility

  • main.js owns several interaction rules directly.
  • HUD.js both renders panels and mutates gameplay systems from button handlers.
  • Environment.js combines zone generation, collision metadata, portal metadata, and station placement.

That mixed responsibility is not fatal for this project size, but it is the first thing that will slow feature work as mechanics multiply.

Persistence

SaveSystem snapshots the live systems into JSON: zone, player position, PP, stats, inventory, pedometer tracks, drones, equipment, game stats, achievements, minigame, ascension, and auto-combat.

Loading applies JSON back into the existing system instances, then returns the zone and player coordinates so main.js can switch zones and teleport the player.

Practical Change Guide

Change balance

Start in config.js. It holds rates, costs, skills, terrain, and unlock thresholds.

Add a new UI panel

Add static shell markup in index.html, styling in CSS, then render/wire it from HUD.js.

Add a new zone

Add zone build logic and metadata in Environment.js, then connect unlocks/portals through config and main loop logic.

Growth Pressure

  • main.js is the coordinator and has accumulated interaction logic.
  • Environment.js builds many zones and props in one large file.
  • HUD.js renders and wires nearly every non-combat panel.

Those three files are the natural refactor targets as the game grows.
